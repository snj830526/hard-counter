# 개발 가이드

## 개발 목표

현재 단계의 최우선 목표는 기능 수를 늘리는 것이 아니라 기본 복싱 조작의 손맛과 시각적 어색함을 줄이는 것이다.

우선순위는 다음과 같다.

1. 입력이 누락되지 않는다.
2. 이동, 공격, 스웨이의 결과를 즉시 이해할 수 있다.
3. 거리와 방향이 히트 판정 및 애니메이션과 일치한다.
4. 타격의 리듬과 피드백이 만족스럽다.
5. 이후에 새로운 공격과 게임 모드를 추가한다.

## 작업 시작 절차

1. `develop`을 최신 상태로 만든다.
2. 작업 목적에 맞는 `feature/<기능명>` 브랜치를 만든다.
3. 사용자에게 보이는 완료 조건을 먼저 정한다.
4. 관련 코드와 문서를 함께 확인한다.
5. 작은 단위로 구현하고 시뮬레이터에서 반복 확인한다.

Git 명령과 브랜치 규칙은 [Git 및 버전 관리](GIT_WORKFLOW.md)를 따른다.

## 구현 원칙

### 전투 규칙과 표현 분리

피해, 상태 전이, 카운터 가능 시간은 `CombatEngine`에서 결정한다. `CombatScene`과 `FighterNode`는 그 결과를 화면과 햅틱으로 표현한다.

3D 표현은 `CombatArena3DRenderer`가 선수 변환과 접촉 좌표를 조정하고, `Arena3DRingNode`와 `Arena3DCameraController`가 링 및 카메라를 각각 소유한다. HUD 생성과 안전 영역 배치는 `CombatHUDNode`가 담당한다. 이 타입들은 피해나 전투 상태 전이를 다시 계산하지 않는다.

### 캐릭터 모션 계층 분리

캐릭터 모션은 다음 방향으로만 의존한다.

```text
CombatEvent
  → FighterNode
      → FighterFullBodyMotionController
          → FighterBodyMotionFrame
              → FighterLocomotionController / FighterFullBodyActionPoseSolver
                  → FighterFullBodyPoseSolver
              → FighterRig
```

- 스틱은 이동 결과가 아니라 이동 의도다. 실제 이동은 지지발, 현재 스텝, 행동 상태와 무게중심이 허용하는 범위에서 `FighterFullBodyMotionController`가 결정한다.
- `FighterBodyMotionFrame`, `FighterPose`와 `FighterLocomotionFrame`은 화면 노드가 아닌 값 데이터로 유지한다.
- 모션 계산 타입에는 `SKNode`를 전달하지 않는다.
- `FighterRig`는 관절 계층을 만들지만 전투 상태나 타격 판정을 알지 못한다.
- `FighterNode`는 계산과 리그 생성 코드를 복제하지 않고 두 계층을 연결한다.
- 실제 이동과 시각적 셔플은 반드시 `FighterBodyMotionFrame.stepProgress` 하나만 사용한다. 표현 계층에 두 번째 보행 시계를 만들지 않는다.
- 스웨이와 공격은 척추 노드의 위치를 골반에서 떼어 이동하지 않는다. 발–무릎–골반–척추–머리–팔의 순서로 공통 전신 솔버를 통과한다.
- 상태 전환은 하체, 몸통, 팔의 반응 속도를 나눈 포즈 관성화를 사용한다. 타격 팔의 속도를 높이기 위해 복귀 몸통까지 같은 속도로 만들지 않는다.
- 보행 포즈가 계산된 뒤 발 접촉점은 실제 화면 이동량을 변환한 IK 목표로 보정한다. IK는 지지발을 우선하되 관절 제한을 넘겨 포즈를 다시 설계할 정도로 강하게 적용하지 않는다.
- 공격 팔은 어깨·팔꿈치 각도를 독립적으로 찍지 않고 기술별 주먹 목표점을 2본 해법으로 푼다. 목표점이나 팔 길이를 바꿀 때 스트레이트의 과신전, 스매시의 머리 위 이탈, 어퍼컷의 얼굴 관통을 60fps 확대 프레임으로 함께 확인한다.

### 튜닝 수치 중앙화

다음 값은 임의의 화면 코드에 직접 넣지 않고 우선 `CombatTuning`에 둔다.

- 공격 시작, 활성, 회복 시간
- 이동 속도와 행동별 이동 배율
- 피해량, 사거리, 최소 선수 간격
- 히트 스톱, 화면 흔들림, 애니메이션 시간

수치 변경은 한 번에 하나의 목적을 가져야 한다. 예를 들어 “입력 반응 개선”과 “펀치 피해 증가”를 같은 튜닝 커밋에 섞지 않는다.

### 프레임 독립 이동

위치 변화는 `deltaTime`을 사용한다. 프레임 저하가 발생해도 이동 속도와 전투 시간이 크게 달라지지 않아야 한다.

### 멀티터치 유지

이동을 유지한 채 펀치 또는 스웨이를 누를 수 있어야 한다. `LocalInputSource`가 이동 터치와 최근 스웨이 방향, 펀치 버퍼를 소유하며 `touchesEnded`와 `touchesCancelled`에서 확실히 해제한다. `CombatScene`은 로컬과 CPU 입력을 모두 `FighterCommand`로 실행하고, 새 입력 방식을 추가할 때 전투 엔진을 직접 호출하는 별도 경로를 만들지 않는다.

### 실제 화면 우선

캐릭터 크기, 안전 영역, 손가락 가림, 햅틱은 실제 iPhone에서 최종 판단한다. 시뮬레이터는 빠른 반복과 화면 구성 확인 용도로 사용한다.

### 근거리 통신 경계

- 로비 화면은 `NearbyLobbyService`의 공개 상태만 관찰하고 `NWConnection`을 직접 다루지 않는다.
- TCP 메시지는 완전한 패킷 단위를 보장하지 않으므로 4바이트 길이 헤더와 JSON 본문으로 프레이밍한다.
- 메시지에는 프로토콜 버전을 넣고 버전 또는 선수 식별자가 맞지 않으면 연결을 종료한다.
- 캐릭터 선택을 바꾸면 준비 상태를 해제해 양쪽 화면의 준비 조건이 어긋나지 않게 한다.
- 전투 입력은 30Hz로 전송하고 로컬 기기에서 즉시 실행해 조작 지연을 숨긴다.
- 호스트가 15Hz로 위치·체력·스태미너·승패를 보내고 게스트가 부드럽게 위치를 보정한다. 판정의 최종 기준은 호스트다.
- 양쪽 준비가 완료되면 호스트가 매치 ID를 만들고 같은 연결로 경기 시작을 알린다.
- KO 후에는 어느 쪽이든 재대결을 요청하거나 요청을 취소할 수 있다. 양쪽이 모두 수락한 경우에만 호스트가 권위 상태를 초기화하고 새 카운트다운을 시작한다.

## 손맛 튜닝 순서

손맛 문제는 아래 순서로 좁힌다.

1. 입력이 들어왔는가
2. 상태 머신이 행동을 허용했는가
3. 이동 또는 공격 수치가 너무 작거나 느린가
4. 애니메이션이 판정보다 늦거나 빠른가
5. 햅틱과 화면 피드백이 타격 순간과 일치하는가

한 번에 여러 수치를 크게 바꾸지 않는다. 변경 전후를 같은 상황에서 비교할 수 있도록 급반전, 이동 펀치, 스웨이 카운터 같은 반복 시나리오를 사용한다.

모션의 역동성을 높일 때 정지 호흡이나 관절 떨림을 키우지 않는다. 이동은 지지발의 하중–골반 이동–상체 후행, 공격은 하체 압축–골반–흉곽–주먹–후속 동작, 스웨이는 빠른 이탈–정점 유지–느린 복귀의 실루엣 차이를 먼저 키운다. 카메라 배율은 이 동작을 판단하기 전에 확정하고, 근거리에서도 머리·장갑·복싱화와 타격 공간이 화면에서 잘리지 않는지 확인한다.

이동 속도와 행동 속도는 별도로 조정한다. 이동은 한 발이 체중을 받고 다른 발이 따라오는 시간을 확보하되, 펀치의 활성 구간과 스웨이 진입 반응은 느리게 만들지 않는다. 발 IK의 바닥 기준은 현재 몸 위치가 아니라 중립 자세에서 캡처한 고정 높이를 사용해 골반 바운스가 발까지 들어 올리지 않게 한다.

## 문서 갱신 규칙

- 구현과 문서가 다르면 구현 완료 전에 문서를 함께 고친다.
- 아직 없는 기능은 현재 기능처럼 서술하지 않는다.
- 폴더나 핵심 타입의 책임이 바뀌면 `PROJECT_STRUCTURE.md`를 갱신한다.
- 브랜치 또는 릴리스 정책이 바뀌면 `GIT_WORKFLOW.md`를 갱신한다.
- 플레이 경험의 방향이 바뀌면 `GAME_CONCEPT.md`를 갱신한다.

## 완료 조건

기능 작업은 다음 조건을 모두 만족할 때 완료한다.

- 요청한 동작이 구현됨
- 관련 회귀 시나리오 확인
- iOS Simulator 빌드 성공
- 실제 iOS 대상 빌드 성공
- `git diff --check` 통과
- 관련 문서 갱신
- feature 브랜치 커밋 및 원격 푸시
