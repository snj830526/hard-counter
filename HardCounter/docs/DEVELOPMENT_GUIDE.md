# 개발 가이드

## 개발 목표

현재 단계의 최우선 목표는 기능 수를 늘리는 것이 아니라 기본 복싱 조작의 손맛과 시각적 어색함을 줄이는 것이다.

우선순위는 다음과 같다.

1. 입력이 누락되지 않는다.
2. 이동, 공격, 스웨이의 결과를 즉시 이해할 수 있다.
3. 거리와 방향이 히트 판정 및 애니메이션과 일치한다.
4. 타격의 리듬과 피드백이 만족스럽다.
5. 이후에 새로운 공격과 게임 모드를 추가한다.

## 작업 시작 절차

1. `develop`을 최신 상태로 만든다.
2. 작업 목적에 맞는 `feature/<기능명>` 브랜치를 만든다.
3. 사용자에게 보이는 완료 조건을 먼저 정한다.
4. 관련 코드와 문서를 함께 확인한다.
5. 작은 단위로 구현하고 시뮬레이터에서 반복 확인한다.

Git 명령과 브랜치 규칙은 [Git 및 버전 관리](GIT_WORKFLOW.md)를 따른다.

## 구현 원칙

### 전투 규칙과 표현 분리

피해, 상태 전이, 카운터 가능 시간은 `CombatEngine`에서 결정한다. `CombatScene`과 `FighterNode`는 그 결과를 화면과 햅틱으로 표현한다.

### 캐릭터 모션 계층 분리

캐릭터 모션은 다음 방향으로만 의존한다.

```text
CombatEvent
  → FighterNode
      → FighterPoseResolver / FighterLocomotionController
          → FighterPose / FighterLocomotionFrame
              → FighterRig
```

- `FighterPose`와 `FighterLocomotionFrame`은 화면 노드가 아닌 값 데이터로 유지한다.
- 모션 계산 타입에는 `SKNode`를 전달하지 않는다.
- `FighterRig`는 관절 계층을 만들지만 전투 상태나 타격 판정을 알지 못한다.
- `FighterNode`는 계산과 리그 생성 코드를 복제하지 않고 두 계층을 연결한다.
- 향후 발 고정 IK와 무게중심 계산은 `FighterLocomotionController` 내부에서 교체한다.

### 튜닝 수치 중앙화

다음 값은 임의의 화면 코드에 직접 넣지 않고 우선 `CombatTuning`에 둔다.

- 공격 시작, 활성, 회복 시간
- 이동 속도와 행동별 이동 배율
- 피해량, 사거리, 최소 선수 간격
- 히트 스톱, 화면 흔들림, 애니메이션 시간

수치 변경은 한 번에 하나의 목적을 가져야 한다. 예를 들어 “입력 반응 개선”과 “펀치 피해 증가”를 같은 튜닝 커밋에 섞지 않는다.

### 프레임 독립 이동

위치 변화는 `deltaTime`을 사용한다. 프레임 저하가 발생해도 이동 속도와 전투 시간이 크게 달라지지 않아야 한다.

### 멀티터치 유지

이동을 유지한 채 펀치 또는 스웨이를 누를 수 있어야 한다. `LocalInputSource`가 이동 터치와 최근 스웨이 방향, 펀치 버퍼를 소유하며 `touchesEnded`와 `touchesCancelled`에서 확실히 해제한다. `CombatScene`은 로컬과 CPU 입력을 모두 `FighterCommand`로 실행하고, 새 입력 방식을 추가할 때 전투 엔진을 직접 호출하는 별도 경로를 만들지 않는다.

### 실제 화면 우선

캐릭터 크기, 안전 영역, 손가락 가림, 햅틱은 실제 iPhone에서 최종 판단한다. 시뮬레이터는 빠른 반복과 화면 구성 확인 용도로 사용한다.

### 근거리 통신 경계

- 로비 화면은 `NearbyLobbyService`의 공개 상태만 관찰하고 `NWConnection`을 직접 다루지 않는다.
- TCP 메시지는 완전한 패킷 단위를 보장하지 않으므로 4바이트 길이 헤더와 JSON 본문으로 프레이밍한다.
- 메시지에는 프로토콜 버전을 넣고 버전 또는 선수 식별자가 맞지 않으면 연결을 종료한다.
- 캐릭터 선택을 바꾸면 준비 상태를 해제해 양쪽 화면의 준비 조건이 어긋나지 않게 한다.
- 전투 입력은 30Hz로 전송하고 로컬 기기에서 즉시 실행해 조작 지연을 숨긴다.
- 호스트가 15Hz로 위치·체력·스태미너·승패를 보내고 게스트가 부드럽게 위치를 보정한다. 판정의 최종 기준은 호스트다.
- 양쪽 준비가 완료되면 호스트가 매치 ID를 만들고 같은 연결로 경기 시작을 알린다. 재시작도 호스트만 요청한다.

## 손맛 튜닝 순서

손맛 문제는 아래 순서로 좁힌다.

1. 입력이 들어왔는가
2. 상태 머신이 행동을 허용했는가
3. 이동 또는 공격 수치가 너무 작거나 느린가
4. 애니메이션이 판정보다 늦거나 빠른가
5. 햅틱과 화면 피드백이 타격 순간과 일치하는가

한 번에 여러 수치를 크게 바꾸지 않는다. 변경 전후를 같은 상황에서 비교할 수 있도록 급반전, 이동 펀치, 스웨이 카운터 같은 반복 시나리오를 사용한다.

## 문서 갱신 규칙

- 구현과 문서가 다르면 구현 완료 전에 문서를 함께 고친다.
- 아직 없는 기능은 현재 기능처럼 서술하지 않는다.
- 폴더나 핵심 타입의 책임이 바뀌면 `PROJECT_STRUCTURE.md`를 갱신한다.
- 브랜치 또는 릴리스 정책이 바뀌면 `GIT_WORKFLOW.md`를 갱신한다.
- 플레이 경험의 방향이 바뀌면 `GAME_CONCEPT.md`를 갱신한다.

## 완료 조건

기능 작업은 다음 조건을 모두 만족할 때 완료한다.

- 요청한 동작이 구현됨
- 관련 회귀 시나리오 확인
- iOS Simulator 빌드 성공
- 실제 iOS 대상 빌드 성공
- `git diff --check` 통과
- 관련 문서 갱신
- feature 브랜치 커밋 및 원격 푸시
